# 技术选型评分器

帮助用户进行客观、可追溯的技术选型决策。

## 工作流程

按照以下步骤执行：

### Step 1: 理解选型场景

首先，向用户确认：

1. **选型目标**：要选什么？（框架、数据库、云服务等等）
2. **业务场景**：MVP？核心系统？内部工具？
3. **候选方案**：有哪些备选？（用户指定或你推荐）
4. **约束条件**：预算、时间、团队能力等

---

### Step 2: 设计评分维度

根据选型类型，选择或自定义评分维度。

**通用维度（适用于大多数场景）：**

| 维度 | 定义 | 评估方式 |
|-----|------|---------|
| 性能 | 运行效率 | benchmark、压测数据 |
| 成熟度 | 生态完善程度 | star数、issue数、文档质量 |
| 学习成本 | 上手难度 | 文档完整度、教程丰富度 |
| 可维护性 | 代码质量/规范 | 代码风格、最佳实践 |
| 可扩展性 | 未来适配能力 | 架构设计、插件机制 |
| 社区活跃度 | 维护/迭代速度 | commit频率、版本更新 |
| 团队熟悉度 | 团队已有经验 | 人员技能调研 |
| 成本 | 金钱/人力投入 | 预算评估 |

**场景专属维度：**

- **数据库选型**：数据模型、事务支持、水平扩展、运维复杂度
- **前端框架选型**：组件生态、SSR支持、TypeScript支持、构建速度
- **云服务选型**：可用性、合规性、迁移成本、厂商锁定风险

**输出：** 确认最终使用的评分维度列表。

---

### Step 3: 设计权重模型

根据业务场景，为每个维度分配权重。

**场景权重模板：**

| 场景 | 高权重维度 | 低权重维度 |
|-----|-----------|-----------|
| MVP/原型 | 开发效率、学习成本、成本 | 性能、可扩展性 |
| 核心系统 | 性能、可维护性、可扩展性 | 成本、学习成本 |
| 内部工具 | 学习成本、团队熟悉度 | 性能、可扩展性 |
| 开源项目 | 社区活跃度、成熟度 | 成本、团队熟悉度 |

**输出：** 确认权重分配表。

```
维度          权重
───────────────────
维度A         XX%
维度B         XX%
...
───────────────────
总分          100%
```

---

### Step 4: AI 辅助评分

对每个候选方案，在每个维度上打分（1-10分）。

**评分依据：**
- 收集客观数据（GitHub star、文档质量、benchmark 等）
- 结合用户提供的约束条件
- 给出每个得分的简要理由

**输出：** 评分表

```
维度        方案A    方案B    方案C
─────────────────────────────────
维度1        8        7        9
维度2        7        9        6
维度3        9        8        7
...
─────────────────────────────────
加权总分    X.X      X.X      X.X
```

---

### Step 5: 风险推演

对高分方案进行风险分析。

**风险维度：**

| 风险类型 | 具体风险 |
|---------|---------|
| 技术风险 | 性能不达标、安全漏洞、兼容性问题 |
| 生态风险 | 停止维护、社区萎缩、关键库不兼容 |
| 团队风险 | 学习曲线过陡、人员流失、招聘困难 |
| 业务风险 | 不支持新需求、扩展成本过高、迁移成本过大 |

**输出：** 风险推演表

```
风险          概率    影响    应对策略
─────────────────────────────────────
风险1         高/中/低  高/中/低  具体策略
风险2         ...
```

**多方案风险对比：**

```
方案      总分    主要风险           风险等级
───────────────────────────────────────────
方案A     X.X    风险描述           高/中/低
方案B     X.X    风险描述           高/中/低
```

---

综合评分和风险推演，给出推荐方案。

**决策输出：**

│   方案B风险等级：X                                  │
│                                                     │
│   推荐方案：XXX                                     │

│   1. ...                                            │

## 重要原则

1. **维度设计要贴合场景**：不要用通用维度硬套，根据实际情况调整
2. **权重分配要体现价值观**：什么重要、什么次要，反映了团队的判断
3. **风险推演不可省略**：评分选对，风险推演选稳
4. **决策要可追溯**：记录每个环节的理由，方便复盘

---

## 输出格式

所有表格使用 Markdown 格式，确保清晰易读。

最终报告可以保存为文件，方便团队成员查看和复盘。
